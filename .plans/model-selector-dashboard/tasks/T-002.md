# T-002: Frontend — Estado y eventos de modelo

## Objetivo
Agregar al frontend la capacidad de rastrear el modelo activo por sesión y manejar el evento SSE `model_select`.

## Archivos a modificar

### 1. `packages/dashboard/frontend/src/api-types.ts`
- Agregar tipos para las nuevas APIs:
  ```ts
  export interface ModelInfo {
    provider: string;
    id: string;
    name: string;
    api: string;
    reasoning: boolean;
  }
  ```

### 2. `packages/dashboard/frontend/src/store/runtime.ts`
- Agregar campo `model?: ModelInfo` a `SessionEntry`
- Inicializarlo como `undefined` al crear sesiones

### 3. `packages/dashboard/frontend/src/store/atoms.ts`
- Agregar selector/atom para leer el modelo de una sesión (si no está cubierto por el mapa de sesiones)

### 4. `packages/dashboard/frontend/src/store/hooks.ts`
- Agregar `useSessionModel(sessionId: string | null): ModelInfo | undefined`

### 5. `packages/dashboard/frontend/src/store/actions.ts`
- Agregar métodos al `Actions` object:
  - `async loadAvailableModels(): Promise<ModelInfo[]>` — llama `orpc.sessions.models.list({})`
  - `async setSessionModel(sessionId, provider, modelId): Promise<void>` — llama `orpc.sessions.models.set({...})`

### 6. `packages/dashboard/frontend/src/store/event-handlers/index.ts`
- Remover `"model_select"` del `Set` `explicitNoOps`
- Agregar handler `handleModelSelect` que actualice `entry.model` en el store para la sesión correspondiente
- El evento `model_select` probablemente tiene payload con `provider`, `id`, `name`

### 7. `packages/dashboard/frontend/src/store/event-handlers/types.ts` (si existe)
- Verificar que el tipo `ServerEvent` incluye `model_select` con el payload correcto

## Notas
- El estado de modelos disponibles puede mantenerse localmente en el componente selector (no necesita ser global) para simplificar
- Pero el modelo actual de la sesión SÍ debe estar en el store global porque se muestra en el header
