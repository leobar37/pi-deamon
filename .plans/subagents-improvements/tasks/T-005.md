# T-005: Split LionRuntime

## Objective
Extract concerns from the `LionRuntime` god object (~568 lines, 28 methods, 7+ concerns) into focused modules.

## Problem
`LionRuntime` (src/lion/runtime.ts) handles:
1. Persistence (save/restore state)
2. State transitions (active/inactive, planning/building)
3. Run tracking (startRun, finishRun, recordSubagentResult)
4. Subagent job tracking (#subagentJobs)
5. Subagent UI tracking (#subagentUi)
6. Dashboard lifecycle (start/stop)
7. Logging (logState, logTool, logError)
8. Main session bridge
9. Controller management
10. Retention/release

## New Module Structure

### `src/lion/persistence.ts`
- Extract `LionPersistence` class from LionRuntime (runtime.ts lines 26-53 + save/restore methods)
- Methods: `saveState(ctx, state)`, `restoreState(ctx) -> LionState | null`, `saveCore(ctx, core)`, `restoreCore(ctx) -> PersistedLionCoreState | null`

### `src/lion/orchestrator.ts` (or `src/lion/runner.ts`)
- Extract run tracking from LionRuntime (core.ts is already separate but runtime wraps it)
- Actually, `core.ts` already handles LionCore. The runtime wraps it. Keep Core as-is.
- Extract `LionJobTracker`: subagent job lifecycle (startJob, finishJob, recordEvent, retainSubagent, releaseRun)

### `src/lion/logger.ts`
- Extract logging methods (logState, logTool, logError)
- Simple utility: format and push to session entries

### Keep in runtime.ts
- State transitions (activate, deactivate, isActive, etc.)
- Dashboard lifecycle (startDashboard, stopDashboard - thin delegation)
- Controller management (ensureController, createSubAgentController)
- Logger + Persistence + JobTracker instantiation and wiring

## Files Affected
- New: `src/lion/persistence.ts`
- New: `src/lion/orchestrator.ts` (job tracker)
- New: `src/lion/logger.ts`
- Modify: `src/lion/runtime.ts` (delegate to new modules, keep orchestration glue)
- Modify: `src/lion/index.ts` (update exports)
- Modify: test files that reference runtime methods (workflow.test.ts)

## Acceptance Criteria
- `runtime.ts` is no more than ~200 lines
- `persistence.ts`, `orchestrator.ts`, `logger.ts` each focus on one concern
- All existing functionality preserved
- All existing tests pass

## Risk
- Medium risk: refactoring runtime.ts touches the most tested code path (workflow.test.ts)
- Mitigation: one extraction at a time, run tests after each

## Validation
1. Run lion workflow test after each extraction
2. Verify all state transitions work
3. Verify persistence roundtrip
