# T-001: UI Provider con Jotai

## Objetivo
Crear un `UIProvider` basado en Jotai para manejar el estado global de UI, reemplazando el Zustand store actual (`ui.ts`) que nunca se usa como provider.

## Archivos a crear

### `src/store/ui/atoms.ts`
Atoms globales para UI state:

```typescript
import { atom, useAtom } from "jotai";

export const activeSessionIdAtom = atom<string | null>(null);
export const sidebarCollapsedAtom = atom<boolean>(false);
// Preparado para futuros estados:
// export const toastsAtom = atom<Toast[]>([]);
// export const settingsPanelOpenAtom = atom<boolean>(false);
```

### `src/store/ui/hooks.ts`
Hooks tipados para consumir UI state:

```typescript
export function useActiveSession(): string | null
export function useSetActiveSession(): (id: string | null) => void
export function useSidebarCollapsed(): boolean
export function useToggleSidebar(): () => void
```

### `src/store/ui/index.ts`
Re-export público de atoms y hooks.

## Archivos a modificar

### `src/store/ui.ts` → DELETE (o mantener como wrapper deprecated)
- El nuevo sistema vive en `src/store/ui/`
- El archivo `ui.ts` puede eliminarse una vez migrado

### `src/store/provider.tsx` o `src/store/ui/provider.tsx`
- Crear `UIProvider` componente:
  ```tsx
  export function UIProvider({ children }: { children: React.ReactNode }) {
    return <JotaiProvider>{children}</JotaiProvider>;
  }
  ```
  Nota: Como los atoms viven en el Jotai store global (no el de SessionRuntime), necesitamos un provider separado o usar el store por defecto de Jotai.

### `src/App.tsx`
- Reemplazar uso de Zustand `useUIStore` por hooks Jotai
- Envolver con `UIProvider`
- Sincronizar `activeSessionId` con hash routing:
  ```typescript
  function useSyncHashToUI() {
    const sessionId = useHashSessionId();
    const setActive = useSetActiveSession();
    useEffect(() => {
      setActive(sessionId);
    }, [sessionId, setActive]);
  }
  ```

### `src/components/Sidebar.tsx`
- Reemplazar `navigateToSession` directo con `useSetActiveSession`
- El `activeSessionId` debe venir de UI provider, no del prop

### `src/components/ChatView.tsx`
- El `sessionId` prop puede seguir existiendo pero sincronizado

## Notas de diseño
- Los atoms de UI usan el store por defecto de Jotai (no el store de SessionRuntime)
- No mezclar con los atoms del runtime de sesión
- El `UIProvider` debe estar por fuera de `SessionRuntimeProvider` o al mismo nivel
- Los hooks deben ser puramente derivados de atoms — sin lógica de negocio
- Mantener compatibilidad: el hash routing debe seguir siendo fuente de verdad para la sesión activa

## Criterios de éxito
- [ ] `useActiveSession()` retorna la sesión activa
- [ ] `useSetActiveSession()` cambia la sesión activa y actualiza el hash
- [ ] `useSidebarCollapsed()` retorna estado del sidebar
- [ ] `useToggleSidebar()` cambia el estado
- [ ] Hash routing sigue funcionando
- [ ] `bun run build` pasa sin errores
- [ ] Sin regresiones en navegación de sesiones
