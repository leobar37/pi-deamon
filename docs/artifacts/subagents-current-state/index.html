Subagentes: Estado Actual del Sistema

Mapa de como funcionan los subagentes hoy en pi-mono: la extension legacy subagents esta deshabilitada y todo el flujo de delegacion vive dentro de Lion, como orquestador de planeacion, ejecucion y revision con contratos estructurados de evidencia.

i

La extension packages/core/src/extensions/subagents/index.ts es ahora no-op por diseño: Lion ya provee delegacion estructurada (executor, reviewer, validator, etc.) con orquestacion basada en planes. 1

Panorama general

El usuario emite un slash command. La sesion principal de coding-agent tiene inyectado Lion como ExtensionAPI. LionRuntime mantiene el estado y delega a TaskRunner, quien arma un ExecutionPlan. TaskExecutor lo despacha respetando la estrategia (sequential, parallel, chain) y para cada tarea crea un SubAgentInstance contra una SubAgentDefinition registrada. El bus de eventos conecta todo con el frontend React via oRPC/SSE.

2

Definiciones builtin

core/definitions/index.ts exporta BUILTIN_DEFINITIONS que se cargan al iniciar Lion. Cada definicion es una SubAgentDefinition inmutable con capacidades, modelo por defecto y fallbacks, nivel de pensamiento y un instructionBuilder que decora el system prompt.

Definition Capabilities Modelo Fallback Thinking Tools Rol
executor edit · write · execute kimi-coding/kimi-for-coding deepseek/deepseek-v4-pro high verbose Implementa cambios concretos
planner solo research kimi-coding/kimi-for-coding deepseek/deepseek-v4-flash medium read · glob · grep · bash Produce plan decision-complete
analyzer solo research deepseek/deepseek-v4-flash — low read · glob · grep · bash Mapea codigo, riesgos, evidencia
reviewer bash (no edit) deepseek/deepseek-v4-pro kimi-coding/kimi-for-coding medium read · glob · grep · bash + skill code-review Review con evidencia y false-positives

Hay tambien el validator como rol virtual en runtime (apunta a reviewer sin skill), manejado por el TaskRunner cuando infiere el role desde la definicion. Ver inferRoleFromDefinition en task-runner.ts.

3

Estrategias Lion y phase policy

El estado de Lion combina strategy (none | simple | plan | review) con phase (planning | building). El TaskRunner.applyPhasePolicy aplica esta matriz antes de mandar cualquier tarea, forçando capabilities y tools por contexto. Esto evita que la fase de planning ejecute cambios o que un ejecutor lea plan files en simple mode.

Simple

building

Sin plan durable. decorateTaskPrompt inyecta <lion_context mode="simple"> ligero.

  • executor tareas ad-hoc
  • analyzer/planner/reviewer bajo demanda
  • Sin .plans/ requerido

Plan

planning building

Plan durable bajo .plans/<slug>/ con context.md, requirements.md, task-index.md, checklist.json y tasks/*.md.

planning → solo analyzer/planner/reviewer/validator, read-only

building → source: "active_plan_next_task" ejecuta el siguiente pendiente (executor role) y graba resultado via subagent_record_result

Review

planning building

Plan durable bajo .reviews/ con la skill code-review cargada en cada tarea.

planning → analyzer + planner (read-only)

building → reviewer + validator; executor prohibido; editing deshabilitado

4

Flujo del tool lion_tasks

Cada llamada a lion_tasks pasa por seis pasos: seleccion del origen (plan o tareas explicitas), decoration del prompt segun estrategia, phase policy que ajusta capabilities y tools, construction del ExecutionPlan, dispatch por el ejecutor y finalmente el completion gate que exige subagent_record_result con un verificationStatus (verified | blocked | retryable | unverified) antes de marcar el checklist como "complete".

5

Lifecycle de un SubAgentInstance

SubAgentState emitido por el bus. El frontend mapea a LionVisibleSubagentStatus con un "stalled" intermedio.
6

Visualizacion: dashboard + frontend

Electron arranca el backend core (pi --web) como child process, parsea la URL del dashboard y la expone por IPC al renderer. El canvas React muestra iframes que cargan /thread/<threadId> contra el core backend (Bun HTTP + oRPC + SSE). La SubagentListPanel refleja los SubAgentEvents en tiempo real via useSubAgentStore, y los the detail views consumen AgentRunSidebar + TaskSidebarSection. El dashboard Electron nunca ejecuta sesiones por si mismo, solo renderiza el canvas.

7

Event bus y trazabilidad

SubAgentEventBus emite SubAgentEvents que llegan al SubagentJobManager y al MainSessionBridge. Hay dos loggers paralelos para trazabilidad:

RunLogger

core/lion/run-logger.ts. Heartbeat + run-level log. Persiste tasks por run, actualiza contadores de completed/failed/pending. Recibe start_job, finish_job y eventos completos de cada subagente.

LionLogger

core/lion/logger.ts. Captura state transitions y errores del runtime, incluye LionRuntimeEventBus con eventos lion.* (mode.changed, tasks.start, checklist.snapshot, etc.) consumidos por UI y API.

Limitaciones conocidas del estado actual

  • Stall markStalledJobs marca como stalled tras 2 minutos sin actividad. Causa comun: ejecucion esperando una promesa externa o CWD cambiado.
  • Composite El UI widget (subagents-widget) solo muestra los 4 mas recientes. Para +50 jobs simultaneos usar el frontend en drawer.
  • Gate El completion gate obliga a subagent_record_result. Si un executor termina sin registrar, hay un solo retry automatico antes de marcar retryable.
  • Cleanup El SubAgentController mantiene maximo 50 instancias en memoria; las viejas (>30 min completadas) se eliminan automaticamente.
  • Bridge Solo el thread principal (main session) es controlable via RPC. Subagentes son steer-only desde el sidebar.
